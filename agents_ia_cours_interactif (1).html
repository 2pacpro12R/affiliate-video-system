Cours interactif • Architecture des agents IA

Comprendre les agents IA en profondeur

Un agent IA n’est pas juste un modèle qui répond à une question. C’est un système orienté objectif qui perçoit un contexte, raisonne, choisit des outils, mémorise l’état, agit, observe les résultats, puis corrige sa trajectoire.

Vision centrale

Formule mentale à retenir

Agent IA = Objectif + Contexte + Raisonnement + Outils + Mémoire + Contrôle
Un chatbot “parle”. Un agent “travaille” : il découpe la tâche, exécute des étapes et vérifie ce qu’il a fait.

01 — Intro

Qu’est-ce qu’un agent IA ?

Définition profonde, différence avec un LLM simple et boucle perception → décision → action.
Définition

Un système orienté mission

Un agent IA reçoit un but, construit un état interne, choisit une stratégie, exécute des actions, puis évalue si l’objectif est atteint.

Différence

LLM ≠ Agent

Le LLM est le moteur de compréhension et de génération. L’agent est l’assemblage complet : modèle + mémoire + outils + orchestration + garde-fous.

Idée clé

Un agent boucle

Il ne s’arrête pas forcément à une réponse unique. Il peut itérer, corriger, demander un autre outil, ou replanifier selon les résultats obtenus.

Boucle cognitive

La séquence canonique d’un agent

PercevoirEntrées, contexte, données
ComprendreIntention, contraintes
PlanifierSous-tâches, ordre
AgirOutils, API, calcul
ObserverRésultats obtenus
ÉvaluerCorrect ? Complet ?
RéviserReplanifier ou conclure
Erreur fréquente

“Un agent, c’est juste un gros prompt”

Non. Un prompt seul ne gère ni état robuste, ni permissions, ni mémoire durable, ni exécution de workflows réels.

La vraie différence industrielle se voit dans l’orchestration, les outils, les contrôles et l’observabilité.
Analogie

Mini-entreprise logicielle

Le modèle joue l’analyste, la mémoire tient les dossiers, les outils exécutent les tâches, et l’orchestrateur coordonne le tout comme un manager. 

LLM        -> raisonne et formule
Mémoire    -> conserve état et contexte
Outils     -> agissent sur le monde
Orchestrateur -> séquence et contrôle
02 — Concepts

Les composants fondamentaux

Tout agent sérieux repose sur des briques bien séparées, chacune ayant un rôle précis.
Bloc 1

Objectif

Ce que l’agent doit accomplir : répondre, analyser, organiser, comparer, surveiller ou agir sur un système.

  • Peut être explicite ou implicite
  • Doit être contraint par un résultat attendu
  • Peut inclure budget, délai, sécurité
Bloc 2

Contexte / Perception

Les entrées qui nourrissent l’agent : texte, documents, API, capteurs, historique, métadonnées, événements.

Le défi n’est pas seulement de lire, mais de transformer ces signaux en un état exploitable.
Bloc 3

Raisonnement

Décider quoi faire, dans quel ordre, avec quel outil, et comment juger la qualité du résultat intermédiaire.

Bloc 4

Planification

Décomposer une mission complexe en étapes plus petites et cohérentes. Peut être explicite (plan visible) ou implicite.

Bloc 5

Outils

Recherche, SQL, CRM, email, navigateur, code, API métier, OCR, agenda. Sans outils, l’agent reste surtout discursif.

Bloc 6

Évaluation

Vérifie cohérence, exhaustivité, conformité, qualité, sécurité, et décide s’il faut réessayer ou conclure.

Mémoire

Les 4 formes à connaître

  • Mémoire courte : contexte de la session en cours
  • Mémoire de travail : résultats intermédiaires de la tâche
  • Mémoire longue : préférences et faits persistants
  • Mémoire vectorielle : récupération sémantique de documents via embeddings
RAG

Retrieval-Augmented Generation

Le modèle n’invente pas à partir de ses seuls paramètres : il récupère d’abord des documents, puis génère une réponse ancrée dans ces sources. 

Question -> Recherche documentaire -> Sélection
-> Injection de contexte -> Génération -> Vérification
03 — Architecture

L’architecture interne d’un agent IA

L’agent est un système distribué de décision. Le LLM n’est qu’une pièce de la machine.

Orchestrateur

Le vrai chef d’orchestre

Gère les transitions d’état, décide quand appeler le LLM, quel outil utiliser, combien d’itérations autoriser, et quand arrêter la boucle.

État

State management

Un agent robuste doit stocker un état structuré : objectif, plan courant, résultats, erreurs, permissions, traces et validations humaines éventuelles.

Contrôle

Guardrails

Permissions, validation humaine, limites de coût, domaines autorisés, schémas de sortie, règles métier, filtrage et audit.

Niveau 1

Architecture cognitive

Comment l’agent “pense” :

  • interprétation de l’objectif
  • décomposition
  • sélection d’outil
  • auto-vérification
Niveau 2

Architecture système

Comment l’agent tourne réellement :

  • API, bases, files de messages
  • mémoires et caches
  • workers d’exécution
  • monitoring et logs
Niveau 3

Architecture de gouvernance

Qui contrôle quoi, avec quelle traçabilité, quelle validation, quel audit et quelle conformité.

En entreprise, les agents utiles sont rarement ceux qui “raisonnent le plus”. Ce sont ceux qui raisonnent assez tout en restant contrôlables.
Trade-off clé

Autonomie vs contrôle

Plus l’agent est autonome, plus il peut être productif — mais aussi coûteux ou risqué si mal encadré.

Autonomie ↑  => initiative ↑, vitesse ↑, risque ↑
Contrôle ↑   => sécurité ↑, audit ↑, friction ↑
04 — Patterns

Les architectures et patterns modernes

Tous les agents ne se ressemblent pas. Voici les principaux designs à connaître.
Pattern A

Réactif

Une demande courte, peu d’étapes, décision quasi immédiate. Idéal pour des tâches simples et rapides.

Pattern B

Plan-and-Execute

Le système génère un plan global, puis exécute chaque étape. Plus stable pour les tâches longues.

Pattern C

Tool-Using Agent

Le modèle sélectionne des outils selon les besoins : web, SQL, email, navigateur, calcul, code, API.

Pattern D

Multi-agent

Plusieurs agents spécialisés collaborent : chercheur, planificateur, rédacteur, critique, exécuteur.

ReAct

Reason + Act

Le système alterne raisonnement et action. Il pense, agit, observe, puis pense à nouveau. 

Pensée -> Action outil -> Observation -> Nouvelle pensée -> Conclusion
State Machine

Machine à états

Très utilisée en production. Chaque étape est explicite : intake, retrieval, validation, execution, review, finalization.

C’est souvent le meilleur choix pour la robustesse métier, car chaque transition est contrôlable.
Mono-agent

Simple, moins cher, plus maintenable

Convient à la majorité des cas d’usage réels tant que le domaine est bien cadré.

Multi-agent

Puissant, mais souvent sur-vendu

Intéressant si les rôles sont réellement spécialisés. Sinon, on ajoute surtout de la latence, du coût et de la complexité de coordination.

05 — Exemples

Cas concrets d’architecture d’agents

Voir un agent comme un pipeline vivant permet de mieux comprendre sa valeur réelle.
Exemple 1

Agent RH de présélection

Mission : lire CV + fiche de poste + règles RH, puis produire un résumé argumenté.

  • Extraction de compétences, expériences, signaux clés
  • Matching avec critères obligatoires / souhaitables
  • RAG sur politiques RH internes
  • Scoring, justification, demande de validation humaine
Entrées -> Parsing -> Matching -> Vérification politiques
-> Score -> Résumé -> Validation humaine
Exemple 2

Agent support client

Mission : comprendre un ticket, retrouver la solution, répondre ou escalader.

  • Classification du problème
  • Recherche de KB via RAG
  • Suggestion d’action et réponse contextualisée
  • Escalade si risque, ambiguïté ou action sensible
Ticket -> Intent -> Retrieval -> Réponse / Escalade -> Journalisation
Exemple 3

Agent finance / contrôle

Détecte des anomalies dans des factures ou transactions en croisant historiques, règles et seuils d’alerte.

Ici, l’architecture doit être plus déterministe : règles + scoring + justification + validation humaine.
Exemple 4

Agent personnel de planification

Coordonne agenda, email, préférences et contraintes pour préparer un déplacement ou une journée de travail.

Très utile, mais très sensible : besoin fort de permissions, sandboxing et revue avant actions externes.
Pseudo-code mental

À quoi ressemble la boucle d’un agent ?

state = load_context()
goal = parse_user_request()

while not done:
    plan = update_plan(goal, state)
    action = choose_best_action(plan, state)
    result = execute(action)
    state = integrate(result, state)
    verdict = evaluate(goal, state)

    if verdict == "complete":
        done = True
    elif verdict == "unsafe":
        request_human_review()
        break
    else:
        continue

return final_response(state)

07 — Cheatsheet

Cheatsheet des termes clés

Résumé compact pour révision rapide.

Agent IA

Système autonome ou semi-autonome orienté objectif, capable de raisonner, agir et s’ajuster.

LLM

Grand modèle de langage utilisé pour comprendre, générer, résumer et guider les décisions.

Orchestrateur

Composant qui coordonne étapes, outils, état, budget et règles d’arrêt.

Planification

Décomposition d’un objectif en sous-tâches ordonnées.

Tool Use

Capacité de l’agent à appeler des outils externes : API, SQL, navigateur, email, code.

State

État courant de la tâche : objectif, plan, résultats, erreurs, validations.

Mémoire courte

Contexte temporaire de la session courante.

Mémoire longue

Préférences et informations persistantes réutilisables entre sessions.

Mémoire vectorielle

Index sémantique pour retrouver des documents proches d’une requête.

RAG

Génération augmentée par récupération de documents pertinents.

Guardrails

Garde-fous : permissions, validation humaine, filtres, schémas, limites d’itération.

ReAct

Pattern alternant raisonnement et actions outillées.

State Machine

Architecture à états explicites, robuste pour la production.

Mono-agent

Un seul agent central gère le flux principal.

Multi-agent

Plusieurs agents spécialisés collaborent sur une même mission.

Human-in-the-loop

Validation humaine insérée avant ou après une action sensible.
Résumé final : un agent IA n’est pas seulement un “modèle intelligent”, mais une architecture complète de décision et d’action. En production, la qualité d’un agent dépend autant de son orchestration et de ses contrôles que de la qualité du modèle.
